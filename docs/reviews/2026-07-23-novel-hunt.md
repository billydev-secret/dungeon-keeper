## Method note

The first review's tracks were organized around code health (async hygiene, god modules, migrations, embed style, a11y, mobile) plus per-feature UX deep passes. This second sweep deliberately picked up the attack angles that pass **did not have**, several of which Addendum 3 of the main report explicitly listed as never having run before its quota cutoff:

- money-flow race analysis (beyond the casino/raffle/wager/bounty set already covered)
- systematic JS ↔ API contract checking (request shape, response shape, id serialization)
- Discord hard-limit analysis (25-option selects, 25-field embeds, 1024-char field values, 2000-char content, the 3-second interaction budget)
- exhaustive command-reference drift (shipped copy vs. the registered command tree)
- cross-guild IDOR (a gated route that never scopes its query to the active guild)
- restart / mid-flight state machines (persistent-view re-registration, recoverers, commit ordering across `await`)
- time and boundary logic (period rolls, trailing windows, retention vs. read window, timezone sourcing)
- content injection (`allowed_mentions`, markdown/code-span escaping, nickname-sourced text)
- schema ↔ code drift (in-code `init_*` helpers as an unmigrated second schema source)
- panel runtime logic (mount/unmount lifecycle, DOM replacement vs. bound listeners, two-way control state)
- line-by-line deep reads of `economy_cog.py`, `economy_quests_service.py`, and `routes/config.py`

**Verification status.** Every **S1/S2** item below survived a 3-vote adversarial verification stage with known-duplicate screening against the main report and its addenda — each verifier independently re-read the cited code, attempted to refute the failure scenario, and grepped the prior report for the finding *and its root cause*. Several were reproduced empirically (live TestClient round-trips, temp pytest repros against migrated DBs, `PRAGMA index_list` on a freshly-migrated database). Where verifiers disputed the reported severity, that disagreement is recorded in the finding row rather than silently resolved.

**S3/S4 items are unverified** — single-pass, cited but not adversarially re-checked. Treat them as leads, not conclusions.

Five findings were reported independently by two different hunters (noted inline). Line numbers and file paths were corrected against the working tree where the original citation drifted.

## What came up clean

Several angles were run to completion and produced nothing worth reporting — and that absence is itself signal, because it bounds where the remaining risk lives.

**Money-flow race analysis beyond Addendum 3's set** found no new read-check-write defects: the claim-`UPDATE` pattern in `place_roulette_bet` is exemplary, `process_conversion` is properly idempotent per `(guild, user, local_day)`, and `rent_perk`'s duplicate guard is a real DB constraint rather than a check-then-insert (the role-icon defect above is a *key mismatch*, not a race, and reproduces single-threaded). **Select-option 25-caps** are handled correctly nearly everywhere — `QuestDetailSelect`, `QuestRerollSelect` and the AMA candidate lists all slice `[:25]`; MLT and `QuestClaimSelect` are the two isolated misses, not a pattern. **Parameter-signature drift** in README was spot-checked across eleven parameterized commands (`loop`, `queue`, `247`, `bank emoji`, `bank sponsor`, `bank grant`, `voice sleepkick`, `voice knock`, `voice access`, `todo`, `warn`, `revokewarn`) and turned up exactly one mismatch (`/247`), so that class is genuinely isolated rather than systemic. **SQL injection and template escaping** were re-checked incidentally throughout the config and games route reads and remain clean — no raw interpolation, `esc()` applied consistently. And the **request-body models** for the games, announcements, scheduled-games and photo-challenge routers consistently accept ids as `str`, which is why the snowflake defects found here are all on the **response** side; the write paths were correct everywhere they were checked.
